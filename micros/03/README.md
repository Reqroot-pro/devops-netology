# Организация инфраструктуры для микросервисной системы

## Задача 1: Обеспечить разработку

Оптимальным выбором для организации процессов непрерывной интеграции и доставки станет платформа GitLab. На начальном этапе рекомендуется использовать управляемое облачное решение — **Managed GitLab в Яндекс Облаке**. Такой подход обладает рядом преимуществ:

- **Простота развёртывания**: облачный провайдер предоставляет подробные руководства по настройке инфраструктуры, включая автоматизацию через Terraform. Это позволяет управлять окружением декларативно, реализуя подход «инфраструктура как код».
- **Гибкость при масштабировании**: по мере роста проекта ресурсы (количество и характеристики ВМ для GitLab Runners) легко корректируются с помощью инструментов автоматизации.
- **Встроенная система контроля версий**: GitLab включает собственный репозиторий для хранения исходного кода. При необходимости публикации кода для внешних команд или открытого доступа доступна прямая синхронизация с GitHub.
- **Универсальность запуска пайплайнов**: GitLab CI поддерживает как автоматический запуск сборок по событиям (push, merge request), так и ручной запуск с передачей параметров через веб-интерфейс.
- **Встроенный анализ кода**: платформа содержит средства статического анализа без необходимости подключения внешних систем, таких как SonarQube (в отличие от Jenkins).
- **Защищённое хранение секретов**: для конфиденциальных данных предусмотрен отдельный механизм с контролем доступа.

### Сравнительная таблица решений

| Критерий                     | Jenkins                     | GitLab CI/CD                |
|------------------------------|-----------------------------|-----------------------------|
| Лицензирование               | Открытый исходный код       | Открытый исходный код       |
| Ключевое преимущество        | Экосистема плагинов (>1000) | Глубокая интеграция с VCS   |
| Техническая поддержка        | Сообщество                  | Официальная поддержка       |
| Установка и настройка        | Требует ручной конфигурации | Упрощённая настройка        |
| Развёртывание                | Только самостоятельное      | Поддержка managed-сервисов  |
| Создание пайплайнов          | Jenkins Pipeline (Groovy)   | .gitlab-ci.yml (YAML)       |
| Мониторинг приложений        | Отсутствует                 | Встроенная функциональность |
| Экосистема                   | Плагины                     | Интеграция в рамках платформы |
| API                          | REST API                    | REST + GraphQL API          |
| Поддержка JavaScript         | Да                          | Да                          |
| Интеграция с инструментами   | Через плагины (Slack, GitHub) | Нативная поддержка (GitHub, Kubernetes) |

## Задача 2: Сбор и анализ логов

Наиболее подходящим решением для централизованного логирования является стек **ELK** (Elasticsearch, Logstash, Kibana) в связке с агентом **Filebeat**.

### Архитектура решения:

- **Elasticsearch** — распределённая система хранения и поиска. Рекомендуется размещать на отдельной ноде из-за высоких требований к оперативной памяти и дисковой подсистеме.
- **Logstash** — конвейер обработки данных, выполняющий фильтрацию, парсинг и нормализацию логов перед отправкой в хранилище.
- **Kibana** — веб-интерфейс для визуализации, поиска и анализа логов с возможностью сохранения поисковых запросов и формирования ссылок для совместной работы.
- **Filebeat** — лёгкий агент, устанавливаемый на хосты с микросервисами. Собирает логи из stdout/stderr, обеспечивает буферизацию при недоступности получателя и минимизирует нагрузку на приложения.

Хотя Filebeat способен отправлять данные напрямую в Elasticsearch, предпочтительнее пропускать логи через Logstash для предварительной обработки и структурирования. В качестве альтернативы Logstash допустимо использование **Vector** — современного агента с высокой производительностью, однако его конфигурация требует больших усилий по сравнению с классическим решением.

## Задача 3: Система мониторинга

Для решения задачи мониторинга инфраструктуры и микросервисов рассматриваются два основных варианта: **Zabbix** и стек **Prometheus + Grafana + Alertmanager**.

### Zabbix
- **Преимущества**: простота установки, «коробочная» поддержка множества типов метрик, встроенный веб-интерфейс, возможность мониторинга логов без дополнительных компонентов.
- **Ограничения**: отсутствие развитого языка запросов для агрегации данных, меньшая гибкость при работе со специфичными метриками микросервисов.

### Prometheus + Grafana + Alertmanager
- **Преимущества**:
  - Модель **pull** — сервер самостоятельно опрашивает эндпоинты `/metrics` сервисов, что снижает нагрузку на клиентские приложения.
  - **PromQL** — мощный язык запросов для сложной агрегации и анализа метрик.
  - Лёгкое масштабирование: вертикальное (увеличение ресурсов) и горизонтальное (федерация, шардирование).
  - Богатая экосистема экспортеров для сбора специфичных метрик (базы данных, очереди сообщений, бизнес-метрики).
  - **Grafana** — гибкая система визуализации с поддержкой шаблонных дашбордов и совместного доступа.
  - **Alertmanager** — продвинутая маршрутизация и группировка алертов.

Несмотря на необходимость настройки нескольких компонентов, стек **Prometheus + Grafana + Alertmanager** предпочтительнее для микросервисной архитектуры. Он лучше соответствует требованиям к агрегации данных и гибкости анализа, а также обеспечивает более эффективное использование ресурсов за счёт модели pull. Для сбора системных метрик хостов рекомендуется использовать **Node Exporter**.
